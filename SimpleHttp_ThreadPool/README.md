# C语言 epoll+线程池，实现一个单反应堆(单reactor的变体)服务器
## 什么是线程池？（餐厅模型）
### 不要每次来客人都临时去大街上雇服务员了！我们服务器一开机，就提前雇好 8 个固定的服务员（工作线程），让他们在后台待命。
### 主线程（迎宾员 epoll）拿到用户的请求后，把它打包成一个任务（Task），扔进**任务队列（Queue）**里。8 个服务员谁闲着，谁就去队列里抢任务做。做完了继续回队列等下一个任务。

现在，你的架构已经正式进化为：
“ 主从分离 Reactor + 任务队列 + 固定线程池模型 ”
这就是 Nginx 的精简版，也是 Java 里大名鼎鼎的 Netty 框架 底层的标准运行机制。

## 为什么“发送缓冲区满 (EAGAIN)”的现象消失了？和线程池有关吗？
之前单线程/每次开新线程时为什么会满？
sendfile 是零拷贝技术，从磁盘读数据发到网卡极快！你的 CPU 处理速度远大于网卡的发送速度。系统内核为每个 cfd 分配的发送缓冲区（一般几十KB到几百KB）瞬间就被 sendfile 塞满了，塞满后就会触发 EAGAIN，让你等一会儿再发。
换成线程池后为什么很少出现了？
有了线程池后，系统里有 8 个工作线程加上 1 个主线程在同时抢夺 CPU 资源（这叫线程上下文切换）。
虽然你感觉不到，但在微秒级别上，你的工作线程在执行 while(offset < size) 循环时，可能会被操作系统“剥夺”一下 CPU 控制权，切去让主线程执行 epoll_wait。就在这极短暂的停顿间隙，网卡底层的硬件已经悄悄把缓冲区里的数据发送到网络上了，缓冲区又空出来了！
所以，不是缓冲区变大了，而是多线程微观上的“走走停停”，正好给了网卡把数据发出去的喘息时间。
(注：如果你未来放一个 2GB 的超大电影去下载，由于网卡带宽总是有极限的，依然还是会触发 EAGAIN 的，你代码里写的 usleep(10000) 防护依然是非常必要且优秀的！)
# 我现在到底是几个 epoll？什么是真正的主从 Reactor？
### A. 判断标准：Reactor 的数量 = epoll 树的数量！
你现在的架构：单 Reactor + 工作线程池 (Single Reactor Multi-Thread)
你的代码里，自始至终只调用了一次 epoll_create，只有1 棵 epoll 树运行在主线程里。
主线程（唯一的 Reactor）既要盯着门口（lfd 负责接客），又要盯着大厅所有的客人（cfd 负责看谁要点菜）。客人点菜了，主线程就把菜单（Task）扔给线程池里的 8 个服务员去做。
### 真正的主从 Reactor (Main-Sub Reactor / Multi-Reactor)：
如果你要升级成纯正的主从 Reactor，你的代码里应该有 9 棵 epoll 树！
主线程（Main Reactor）有 1 棵 epoll 树，只挂 lfd，专门接客；
8 个工作线程（Sub Reactor），每个人手里都有一棵独立的 epoll 树。主线程接到新 cfd 后，直接把它挂到某个工作线程的 epoll 树上。以后这个客人的所有读写，全由那个工作线程自己的 epoll_wait 来独立监听！
B. 单 Reactor + 线程池够用吗？
绝对够用！这已经是极其优秀的高并发架构了。只有当并发量达到十万、百万级别，主线程一个人同时监控所有的 lfd 和 cfd 忙不过来（成为性能瓶颈）时，才需要进化到多 Reactor（如 Nginx、Netty）。
## 只打开一个浏览器，是只有一个 cfd 吗？子线程怎么分配的？
你的直觉非常准：“因为目前不是长连接，我每点击一次，就是建立了一个全新的 cfd！”
在你的代码逻辑下（发完数据立刻 close(cfd) 并且没有发 Content-Length），即使你从头到尾只用了一个 Chrome 浏览器标签页，背后发生的真实情况是这样的：
你输入网址回车： 浏览器向服务器发起第 1 次 TCP 三次握手。主线程 accept 生成 cfd_1。浏览器发送 GET 请求，主线程唤醒**线程池里的【服务员 A】**去处理。发完目录后，服务员 A 关闭了 cfd_1。
你点击了某张照片： 浏览器向服务器发起第 2 次全新的 TCP 三次握手。主线程 accept 生成 cfd_2。浏览器发送 GET 图片请求。主线程唤醒线程池里的【服务员 B】（谁闲着谁去）去处理。发完照片后，服务员 B 关闭了 cfd_2。
你点击返回上一级 (..)： 浏览器向服务器发起第 3 次全新的 TCP 三次握手，生成 cfd_3，交给**线程池里的【服务员 C】**处理并关闭。

总结核心真相：
由于你采用的是 HTTP/1.0 短连接机制，浏览器每一次获取新页面、新图片，甚至偷偷获取那个 favicon.ico 小图标，都是完全独立的新 TCP 连接（全新的 cfd）。
在这个过程中，你的 8 个工作线程就像滴滴司机一样，谁在线程池里空闲，系统就会唤醒谁去接下这个新 cfd 的派单，接完一单就断开，继续回车库（阻塞等待）等下一单。
这就是整个 Web 服务器最真实的底层运作图景！你现在已经把网络 I/O 模型、多线程并发、HTTP 协议这三大后端神功全部串联在一起了！

## 长连接（Keep-Alive）下的浏览器与子线程分配
#### 1、全程只创建一个 cfd 吗？
理论上： 是的。如果是严格串行的访问，你点开网页、点图片，用的都是同一个 TCP 连接（同一个 cfd）。
实际上（浏览器的并发优化）： 现代浏览器非常聪明。当它解析到你的网页里有 10 张图片时，为了加速加载，它不会等第一张图下完再去下第二张。浏览器会默认向同一个服务器同时发起 4~6 个 TCP 长连接（这是浏览器的硬性规定，一般单域名最多 6 个并发连接）。所以，你其实会看到服务器 accept 了大约 6 个 cfd，然后这 6 个 cfd 挂在 epoll 树上，长期待命，帮这个浏览器处理后续的所有点击。
#### 2、一直在 epoll 树上吗？由新读写事件触发？
是的！ 发完一次数据后，我们不调用 close(cfd)，也不把它从 epoll 树上 DEL 掉。它就静静呆在树上。当你再次点击网页上的链接，浏览器顺着这个没断的 TCP 通道发来新的 HTTP 请求，系统网卡收到数据，epoll 树立刻被激活，再次触发 EPOLLIN（读事件）。
#### 3、也只有一个子线程处理吗？
不一定！这是线程池的魅力所在。
假设浏览器用 cfd_1 第一次发请求，主线程把它打包成任务扔进队列，**线程池里的【线程A】**抢到了，处理完后不关连接。
过了 5 秒，浏览器又用 cfd_1 发了第二次请求。主线程再次把它打包扔进队列，这次**线程池里的【线程C】**刚好闲着，可能就是【线程C】去处理。
总结： 长连接复用的是 TCP 通道（cfd），而不是复用某个特定的工作线程。（注意：这正是为什么我们在前面必须加 EPOLLONESHOT 的原因，防止同一个 cfd 的数据还没读完，就被两个不同的线程同时抢走！）
